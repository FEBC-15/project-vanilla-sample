# 4장 이슈 관리

- 프로젝트를 진행하면서 발생하는 다양한 이벤트(버그, 추가해야 할 기능, 아이디어, 질문 등)를 등록하고 관리하는 기능

## 1. 마일스톤

- 마일스톤: 유사한 이슈들을 하나로 모아서 관리
- 개발 목표를 마일스톤으로 만들고 관련 이슈들을 미리 생성하면 이슈들의 Open, Close 상태를 한눈에 관리

### 1.1 마일스톤 생성 실습
- Github Repository > Issues > Milestones > New milestone
  + Title: Sprint 1
  + Due data: 2025-11-14
  + Description: 프로젝트 주제 선정, 요구사항 정의서 작성, 화면 정의서 작성(와이어 프레임), 마크업 완료, 프로토타입 완성
  + `Create milestone`

- Github Repository > Issues > Milestones > New milestone
  + Title: Sprint 2
  + Due date: 2025-11-21
  + Description: 회원/상품/구매 기본 기능 구현
  + `Create milestone`

- 그밖에 필요한 마일스톤 추가

## 2. 레이블

- 이슈나 풀 리퀘스트 생성 시 사용할 레이블 관리
- Github Repository > Issues > Labels
  + 기존 레이블을 수정/삭제하거나 새로운 레이블을 추가해서 관리

### 2.1 일반적인 레이블 카테고리

#### 우선순위(Priority)

- `Priority: High`: 가장 높은 우선순위의 이슈
- `Priority: Medium`: 중간 우선순위의 이슈
- `Priority: Low`: 우선순위가 낮은 이슈

#### 유형(Type)

- `Type: bug`: 버그 수정이 필요한 이슈
- `Type: enhancement`: 기존 기능 개선
- `Type: feature`: 새로운 기능 추가
- `Type: documentation`: 문서와 관련된 이슈
- `Type: refactor`: 코드 리팩토링
- `Type: test`: 테스트 코드 관련 이슈

#### 상태(Status)

- `Status: in progress`: 진행 중인 이슈
- `Status: needs review`: 검토가 필요한 이슈
- `Status: blocked`: 다른 작업에 의해 차단된 이슈

#### 난이도(Difficulty)

- `Difficulty: easy`: 초급 개발자가 해결할 수 있는 이슈
- `Difficulty: medium`: 중간 난이도의 이슈
- `Difficulty: hard`: 고난이도의 이슈

#### 범위(Scope)

- `Scope: frontend`: 프론트엔드 관련 이슈
- `Scope: backend`: 백엔드 관련 이슈
- `Scope: API`: API 관련 이슈
- `Scope: UI/UX`: 사용자 인터페이스 및 경험 관련 이슈

#### 기타(Custom 레이블)

- `Etc: help wanted`: 외부의 도움이 필요한 이슈
- `Etc: wontfix`: 해결되지 않을 예정인 이슈
- `Etc: duplicate`: 중복된 이슈

### 2.2 레이블 생성 실습

## 3. 이슈로 지정하는 작업의 규모

- GitHub에서 이슈를 지정할 때, 하나의 이슈는 독립적으로 처리 가능한 작은 기능 단위가 되어야 하며 작업의 규모는 다음 기준에 따라 나눌 수 있지만, 팀이나 프로젝트의 특성에 맞춰 적절히 조정

- 작업의 독립성
  + 하나의 이슈는 가능한 한 독립적인 작업이어야 함.
  + 다른 작업에 지나치게 의존하지 않고, 개별적으로 완료될 수 있는 단위로 나누는 것이 이상적

- 완료 시간
  + 이슈 하나는 짧게는 몇 시간에서 길게는 며칠 안에 완료할 수 있는 작업이 적당.
  + 일반적으로 1~3일 정도 소요되는 작업 단위로 나누는 것을 고려.
  + 너무 큰 작업이라면 여러개의 이슈로 세분화해서 관리

- 구체성
  + 이슈는 명확하고 구체적인 목표를 포함하도록 작성.
  + 예를 들어, "장바구니" 대신 "장바구니 UI 작성", "장바구니에 상품 담기", "장바구니 상품 수량 수정", "장바구니 상품 삭제" 같은 작은 기능 단위로 나누어야 함.

- 기능적 측면
  + 하나의 이슈는 주로 하나의 기능 또는 버그 수정을 다루도록 지정.
  + 너무 많은 기능을 하나의 이슈로 묶으면 작업 관리가 어렵고, 진행 상황 추적이 힘들어짐.

## 4. 이슈 템플릿

- 이슈 작성시 버그 리포트 같은 경우 버그 현상, 재현 시나리오 등 이슈에 포함되어야 하는 내용을 입력 할수 있는 템플릿

### 4.1 이슈 템플릿 작성 실습
#### 프로젝트 루트에 깃허브 이슈 템플릿 폴더 생성
```sh
mkdir -p ".github/ISSUE_TEMPLATE"
```

#### 이슈 템플릿 폴더에 bug_report.md 파일 작성
```md
---
name: 버그 리포트
title: '[버그] 문제 설명'
---

## 버그 설명

어떤 버그가 발생했는지 명확하게 설명해주세요.

## 재현 방법

버그를 재현하기 위한 단계:

1. '...'로 이동
2. '...'를 클릭
3. '...'로 스크롤
4. 오류 발생

## 기대한 동작

정상적으로 동작했을 때 기대한 결과를 설명해주세요.

## 스크린샷

가능하다면, 스크린샷을 추가해주세요.

## 환경

- OS: [예: Windows 11]
- 브라우저: [예: Chrome 137]
- 기타 관련 정보

## 추가 정보

다른 정보가 있으면 추가해주세요.
```

#### 이슈 템플릿 폴더에 new_feature.md 파일 작성
```md
---
name: 기능 요청
title: '[기능 요청] 원하는 기능 설명'
---

## 기능 요청 사항

어떤 기능을 추가하거나 개선하고 싶은지 설명해주세요.

## 해결 방법

이 기능을 어떻게 구현할지에 대한 간단한 설명을 적어주세요.

## 대안

이 기능의 대체 방안이 있으면 알려주세요.

## 추가 정보

기능 요청과 관련하여 추가적으로 설명할 내용이 있으면 적어주세요.
```

## 5. 이슈 생성
- Github Repository > Issues > New issue
  + 이슈 템플릿이 등록되어 있을 경우 템플릿중 하나를 선택하거나 `Blank issue`를 선택해서 빈 화면으로 시작
  + Add a title: 이슈의 제목을 입력. 간결하고 명확하게 작성하는 것이 좋음
  + Add a description: 이슈의 상세 내용을 입력. 버그 리포트의 경우 재현 방법, 기대 동작 등을 포함
  + 선택 사항
    - Assignees: 이슈를 담당할 팀원을 지정. 여러 명을 선택할 수 있음
    - Labels: 이슈의 유형, 우선순위, 범위 등을 나타내는 레이블을 선택. 여러 레이블을 선택할 수 있음
    - Type: 기본으로 제공되는 이슈 분류
    - Projects: 이슈를 연결할 프로젝트를 선택
    - Milestone: 이슈를 연결할 마일스톤을 선택. 스프린트나 릴리스 단위로 관리할 때 사용

### 5.1 이슈 생성 실습
- Github Repository > Issues > New issue > 기능 요청
  + Add a title: 다크 모드 기능 추가
  + Add a description
    ```md
    ---
    name: 기능 요청
    title: '[기능 요청] 다크 모드 지원 추가'
    ---

    ## 기능 요청 사항
    현재 사이트는 라이트 모드만 지원하고 있어 야간에 사용 시 화면이 너무 밝습니다.  
    사용자 환경 개선을 위해 **다크 모드(Dark Mode)**를 추가해 주세요.  
    자동 모드 전환(시스템 설정에 따라 테마 변경)도 함께 지원되면 좋겠습니다.

    ---

    ## 해결 방법
    - CSS 변수 기반의 색상 테마 구조를 적용합니다.  
    - `prefers-color-scheme` 미디어 쿼리를 활용해 시스템 설정에 따라 자동 전환되게 합니다.  
    - 상단 네비게이션 바에 테마 전환 토글 버튼을 추가합니다.  
    - 선택한 테마는 `localStorage`에 저장하여 재방문 시 유지되도록 합니다.

    ---

    ## 대안
    - 다크 모드 대신 **폰트 대비 조정** 옵션을 제공하는 방안도 검토할 수 있습니다.  
    - 초기에는 메인 페이지에만 다크 모드를 적용한 뒤 점진적으로 확장할 수도 있습니다.

    ---

    ## 추가 정보
    테마 전환 토글 버튼은 public/images/theme.svg 파일로 만들어져 있습니다.  
    추후 다크 모드 배너 이미지 교체도 필요할 수 있습니다.
    ```
  + Assignees: 팀원중 한명 선택
  + Labels: `Priority: Medium`, `Type: feature`, `Difficulty: medium`
  + Milestone: `Sprint 2`
  + `Create`

## 6. 커밋 메시지와 이슈

### 6.1 커밋과 이슈 연결
- 커밋 메시지 입력 시 이슈 번호를 지정하면 커밋과 이슈를 연결시킬 수 있음

  ```
  feat: 다크 모드 추가 완료

  #123
  ```

### 6.2 커밋과 이슈 종료
- 커밋 메시지 입력 시 이슈 번호와 다음 키워드를 같이 사용하면 이슈를 종료 시킬 수 있음

- `close`, `closes`, `closed`, `fix`, `fixes`, `fixed`, `resolve`, `resolves`, `resolved`

  ```
  feat: 다크 모드 추가 완료

  close #123
  ```
